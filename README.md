# Sprint_7 Project

Этот проект разработан для тестирования API учебного сервиса Яндекс.Самокат в рамках финального проекта 7 спринта.

## О проекте

Цель проекта — автоматизированное тестирование API сервиса Яндекс.Самокат с использованием Java, Maven, JUnit 4 и RestAssured. В проекте также используется Allure для генерации отчётов о выполненных тестах.

## Подготовка к тестированию

Прежде чем начать писать тесты, следует выполнить следующие шаги:

1. **Создание Maven проекта**:
   ```
   mvn archetype:generate -DgroupId=com.example -DartifactId=Sprint_7 -DarchetypeArtifactId=maven-archetype-quickstart -DinteractiveMode=false
   ```

2. **Добавление зависимостей** в файл `pom.xml`:
   - JUnit 4
   - RestAssured
   - Allure (для генерации отчётов)

3. **Тестирование вручную** в Postman:
   - Изучите API вручную, используя документацию: [qa-scooter.praktikum-services.ru/docs/](http://qa-scooter.praktikum-services.ru/docs/)

4. **Настройка проекта**:
   - Создайте необходимые классы для тестирования.
   - Настройте окружение для работы с RestAssured и JUnit.

## Описание тестов

### Создание курьера

- **Тесты**:
  - Проверка возможности создания курьера.
  - Проверка запрета создания двух одинаковых курьеров.
  - Проверка обязательных полей и корректности кода ответа.
  - Проверка наличия ошибок при отсутствии обязательных полей или при создании курьера с существующим логином.

### Логин курьера

- **Тесты**:
  - Проверка возможности авторизации курьера.
  - Проверка обязательных полей и корректности кода ответа.
  - Проверка наличия ошибок при неправильном логине или пароле.
  - Проверка ответа системы при авторизации несуществующего пользователя.

### Создание заказа

- **Тесты**:
  - Проверка возможности создания заказа с различными вариантами указания цвета.
  - Проверка корректности тела ответа и наличия трека.
  - Использование параметризации для тестирования различных вариантов создания заказа.

### Список заказов

- **Тесты**:
  - Проверка корректности возвращаемого списка заказов.

### Дополнительные задания

Для выполнения дополнительных заданий также разработаны следующие тесты:

- **Удаление курьера**:
  - Проверка успешного и неуспешного удаления курьера.

- **Принятие заказа**:
  - Проверка корректности принятия заказа с различными вариантами параметров.

- **Получение заказа по номеру**:
  - Проверка возможности получения информации о заказе по его номеру.

## Генерация отчётов Allure

Для генерации отчётов Allure выполните следующие действия после выполнения тестов:

1. Сгенерируйте отчёт Allure:
   ```
   mvn allure:report
   ```

2. Добавьте отчёт в репозиторий (не добавляйте всю папку target):
   ```
   git add -f ./target/allure-results/.
   git commit -m "Add Allure report"
   git push
   ```
## Основное задание

### 1. Нейминг элементов корректный
- Что это означает: Все имена классов, методов, переменных и файлов должны быть понятными и следовать общепринятым стандартам именования.
- Как проверить: Посмотрите в шпаргалку по неймингу (https://mkyong.com/java/java-naming-conventions/), чтобы убедиться, что ваши имена соответствуют правилам.

### 2. В pom.xml нет ничего лишнего
- Что это означает: Убедитесь, что в pom.xml нет лишних зависимостей или плагинов.
- Как проверить: Проверьте, чтобы pom.xml содержал только необходимые зависимости для тестирования и разработки. Пример структуры можно найти здесь (https://maven.apache.org/pom.html).

### 3. Тесты лежат в src/test/java
- Что это означает: Все ваши тестовые классы должны находиться в папке src/test/java, а не в src/main/java.
- Как проверить: Проверьте структуру проекта в вашем IDE или файловой системе.

### 4. Для каждой ручки тесты лежат в отдельном классе
- Что это означает: Создайте отдельные тестовые классы для каждой API ручки (например, CreateCourierTest, LoginCourierTest).
- Как проверить: Убедитесь, что для каждого тестируемого эндпоинта создан свой отдельный класс.

### 5. Написаны тесты на ручку «Создать курьера»
- Что это означает: Напишите тесты для эндпоинта, который позволяет создать нового курьера.
- Как проверить: Проверьте, что в CreateCourierTest есть тесты, которые проверяют создание курьера.

### 6. Написаны тесты на ручку «Логин курьера»
- Что это означает: Напишите тесты для эндпоинта, который позволяет курьеру войти в систему.
- Как проверить: Убедитесь, что в LoginCourierTest есть тесты для проверки процесса логина курьера.

### 7. Написаны тесты на ручку «Создать заказ»
- Что это означает: Напишите тесты для эндпоинта, который позволяет создать новый заказ.
- Как проверить: Убедитесь, что в CreateOrderTest есть тесты для создания заказа.

### 8. Написаны тесты на ручку, которая получает список заказов
- Что это означает: Напишите тесты для эндпоинта, который возвращает список всех заказов.
- Как проверить: Проверьте, что в GetOrdersListTest есть тесты, которые проверяют получение списка заказов.

### 9. Для всех шагов автотестов должна быть использована аннотация @Step
- Что это означает: Все шаги тестов должны быть аннотированы @Step, чтобы отчет был более понятным.
- Как проверить: Проверьте, что вы используете @Step из библиотеки Allure. Пример использования: @Step("Создание нового курьера").

### 10. Сделан отчёт с помощью Allure
- Что это означает: Убедитесь, что вы настроили Allure для генерации отчетов и закоммитили отчет в систему контроля версий.
- Как проверить: Проверьте, что вы настроили Allure и добавили его в pom.xml, а также закоммитили отчет, используя allure-maven плагин.

### 11. В тестах проверяется тело и код ответа
- Что это означает: Ваши тесты должны проверять не только HTTP статус код, но и тело ответа.
- Как проверить: Проверьте, что ваши тесты проверяют как статус код, так и содержимое ответа (например, response.getBody()).

### 12. Все тесты независимы
- Что это означает: Тесты не должны зависеть друг от друга.
- Как проверить: Проверьте, что выполнение одного теста не влияет на выполнение другого.

### 13. Необходимые тестовые данные создаются перед тестом и удаляются после того, как он выполнится
- Что это означает: Создавайте необходимые данные в методе @Before и очищайте их в методе @After.
- Как проверить: Проверьте, что вы используете аннотации @BeforeEach и @AfterEach для подготовки данных и очистки ресурсов.

### 14. В тестах нет хардкода

- Что это означает: Избегайте жестко закодированных значений в тестах.
    - Как проверить: Убедитесь, что тесты используют переменные или конфигурационные файлы для значений, таких как URL, ID и прочее.

### 15. В проекте используется Java 11
- Что это означает: Проверьте, что ваш проект настроен на использование Java 11.
- Как проверить: Проверьте pom.xml на наличие maven-compiler-plugin с source и target установленных на 11.
